# STM32G431RBT6 模板工程软件架构说明

## 1. 文档目标

本文档说明当前 `myBLDC` 仓库的真实软件架构状态，以及 FreeRTOS 接入后各层的职责边界。

本文档基于当前仓库源码，不引用 `docs/` 下现有 PDF 内容。

---

## 2. 当前总体结论

当前工程已经不是裸机 `super loop` 模板，而是一个已经完成以下演进的 STM32G431RBT6 工程骨架：

- 目录结构已整理为 `APP / Module / BSP / HAL-CMSIS` 四层
- FreeRTOS v9 内核已经正式接入 `Libraries/freertos/Source`
- APP 层已经从 `while (1) { app_run(); }` 迁移为 “创建任务并启动调度器”
- BSP 层已经承担中断桥接、HAL 配置、newlib 运行库适配与 FreeRTOS 钩子

它当前的定位可以概括为：

- 一个可编译、可烧录、可调试的 STM32G431 工程
- 一个已经具备 RTOS 基础底座的 BLDC 项目模板
- 一个仍处于“平台先搭稳、业务再逐步填充”的首版架构

---

## 3. 当前分层与依赖关系

### 3.1 分层结构

当前项目可以理解为以下结构：

```text
VSCode / OpenOCD / CMake
        |
        v
project/ + .vscode/
        |
        v
app/ + module/ + bsp/
        |
        v
Libraries/ (FreeRTOS + HAL + CMSIS + startup + system)
```

### 3.2 固定依赖方向

```text
APP -> Module
APP -> BSP
Module -> BSP
BSP -> HAL / CMSIS / FreeRTOS
```

固定约束如下：

- `APP` 可以调用 `Module` 和系统级 `BSP` 接口
- `Module` 只能通过 `BSP` 暴露的接口访问硬件
- `APP` 不直接调用 HAL/LL/寄存器接口
- `Module` 不直接包含 HAL 头文件，也不直接访问寄存器
- `Libraries/` 中的 HAL、CMSIS、FreeRTOS 一律视为第三方底座

---

## 4. 当前目录职责

### 4.1 `app/`

负责应用层的启动编排与任务创建。

当前已落地的职责：

- `main()` 顶层入口
- `app_init()` 上层初始化
- `app_start()` 创建任务并启动调度器
- 任务心跳、堆余量、栈高水位等调试状态维护

### 4.2 `module/`

负责可复用业务模块。

当前状态：

- 已存在 `module/motor/` 作为电机控制模块入口
- `module_motor_init()` 已接入 `app_init()`
- `module_motor_process()` 仍保留，但首版尚未挂到 RTOS 调度路径

### 4.3 `bsp/`

负责底层平台与工程内统一硬件接口。

细分职责如下：

- `bsp/board/`
  板级初始化、系统时钟、`Error_Handler()`
- `bsp/config/`
  HAL 配置与 FreeRTOS 配置
- `bsp/core/`
  IRQ Handler、HAL MSP、FreeRTOS 诊断钩子
- `bsp/driver/`
  GPIO/UART/PWM/ADC/TIM 等项目内部驱动接口
- `bsp/runtime/`
  `syscalls.c`、`sysmem.c` 等 newlib 运行库桥接

### 4.4 `Libraries/`

第三方底座层，当前包括：

- `Libraries/startup/`
- `Libraries/stm32g4/`
- `Libraries/cmsis/`
- `Libraries/hal/`
- `Libraries/freertos/Source/`

FreeRTOS 在本工程中的定位与 HAL/CMSIS 一致，属于底座依赖，不承载用户业务代码。

---

## 5. 当前启动链路

当前工程的启动链路如下：

```text
startup_stm32g431xx.s
  -> Reset_Handler
  -> SystemInit()                              // Libraries/stm32g4/system_stm32g4xx.c
  -> 拷贝 .data / 清零 .bss
  -> __libc_init_array()
  -> main()                                    // app/app_main.c
  -> bsp_init()                                // bsp/board/bsp_init.c
     -> HAL_Init()
     -> bsp_clock_init()
     -> bsp_gpio_init()
     -> bsp_uart_init()
     -> bsp_pwm_init()
     -> bsp_adc_init()
     -> bsp_tim_init()
  -> app_init()                                // app/app_task.c
     -> module_motor_init()
     -> 清零 RTOS 诊断状态
  -> app_start()
     -> 创建 motor / service / monitor 任务
     -> vTaskStartScheduler()
```

这条链路说明：

- C 运行时初始化路径保持标准
- 板级初始化仍统一收敛在 `bsp_init()`
- APP 层不再负责裸机主循环，而是切换为 RTOS 调度入口

## 6. FreeRTOS 接入设计

### 6.1 内核源码组织

FreeRTOS 上游来源固定为：

```text
/home/ssw12/third_party/FreeRTOSv9.0.0/FreeRTOS/Source
```

仓库内落库位置为：

```text
Libraries/freertos/Source
```

仓库保留完整 `Source` 树，但当前 CMake 只编译以下文件：

- `list.c`
- `queue.c`
- `tasks.c`
- `timers.c`
- `event_groups.c`
- `portable/GCC/ARM_CM4F/port.c`
- `portable/MemMang/heap_4.c`

`croutine.c` 不参与当前工程构建。

### 6.2 FreeRTOS 配置入口

当前唯一配置入口是：

```text
bsp/config/FreeRTOSConfig.h
```

首版关键配置固定为：

- `configTICK_RATE_HZ = 1000`
- `configTOTAL_HEAP_SIZE = 10 * 1024`
- `configSUPPORT_DYNAMIC_ALLOCATION = 1`
- `configSUPPORT_STATIC_ALLOCATION = 0`
- `configMAX_PRIORITIES = 6`
- `configMINIMAL_STACK_SIZE = 128`
- `configUSE_PREEMPTION = 1`
- `configUSE_MUTEXES = 1`
- `configUSE_COUNTING_SEMAPHORES = 1`
- `configUSE_TIMERS = 1`
- `configCHECK_FOR_STACK_OVERFLOW = 2`
- `configUSE_MALLOC_FAILED_HOOK = 1`

### 6.3 时基与 SysTick

当前 HAL 与 FreeRTOS 共享 `SysTick`。

`bsp/core/stm32g4xx_it.c` 中的 `SysTick_Handler()` 固定行为为：

```c
void SysTick_Handler(void)
{
  HAL_IncTick();
  HAL_SYSTICK_IRQHandler();

  if (xTaskGetSchedulerState() != taskSCHEDULER_NOT_STARTED)
  {
    xPortSysTickHandler();
  }
}
```

这样可以同时保证：

- 调度器启动前，`HAL_Delay()` 仍可用
- 调度器启动后，FreeRTOS tick 正常推进
- HAL 内部基于 `uwTick` 的 timeout 逻辑不失效

### 6.4 SVC / PendSV 归属

当前 `SVC_Handler` 和 `PendSV_Handler` 不再由 `bsp/core/stm32g4xx_it.c` 提供空实现，而是通过 `FreeRTOSConfig.h` 映射给 FreeRTOS 端口：

- `vPortSVCHandler -> SVC_Handler`
- `xPortPendSVHandler -> PendSV_Handler`

这使得上下文切换完全由 FreeRTOS CM4F 端口接管。

### 6.5 堆与运行库关系

当前工程同时保留两套内存能力：

- FreeRTOS 任务/队列/定时器使用 `heap_4`
- newlib 相关的 `malloc` / `_sbrk()` 仍由 `bsp/runtime/sysmem.c` 负责

这意味着：

- RTOS 内核不依赖 newlib `malloc`
- `syscalls.c` / `sysmem.c` 仍继续保留
- 链接脚本保持不变

### 6.6 HAL 配置中的 `USE_RTOS`

`bsp/config/stm32g4xx_hal_conf.h` 中的：

```c
#define USE_RTOS 0U
```

当前必须保持不变，因为 STM32G4 这版 HAL 在 `stm32g4xx_hal_def.h` 中明确要求 `USE_RTOS == 0U`。这不影响 FreeRTOS 内核在本工程中的实际使用方式。

## 7. APP 层当前接口与任务模型

### 7.1 当前顶层接口

当前 APP 固定接口为：

```c
void app_init(void);
void app_start(void);
```

其中：

- `app_init()` 负责模块初始化与 RTOS 调试状态初始化
- `app_start()` 负责创建任务并调用 `vTaskStartScheduler()`

### 7.2 当前任务骨架

首版固定为 3 个任务：

#### `motor` 任务

- 优先级：`tskIDLE_PRIORITY + 3`
- 栈深：`384`
- 周期：`1 ms`
- 当前职责：只保留主控制任务位置并维护心跳

注意：该任务当前**不调用** `module_motor_process()`，目的是先把 RTOS 平台层稳定下来，再接入真实控制循环。

#### `service` 任务

- 优先级：`tskIDLE_PRIORITY + 2`
- 栈深：`256`
- 周期：`10 ms`
- 当前职责：作为未来协议、故障、系统服务的承载位

#### `monitor` 任务

- 优先级：`tskIDLE_PRIORITY + 1`
- 栈深：`256`
- 周期：`1000 ms`
- 当前职责：
  - 维护 3 个任务的心跳计数
  - 记录任务总数
  - 记录各任务栈高水位
  - 记录当前堆余量与最小历史堆余量

### 7.3 调试状态

`app/app_task.c` 中保留了多组 `volatile` 变量，便于在调试器中直接观察：

- `g_app_motor_task_heartbeat`
- `g_app_service_task_heartbeat`
- `g_app_monitor_task_heartbeat`
- `g_app_rtos_task_count`
- `g_app_*_task_stack_high_water`
- `g_app_rtos_free_heap_bytes`
- `g_app_rtos_min_ever_free_heap_bytes`

## 8. BSP 层当前中断与诊断设计

### 8.1 IRQ 归属

固定原则如下：

- 向量表继续保留在 ST 启动文件
- 具体中断函数统一放在 `bsp/core/`
- 中断中的 HAL 调用由 BSP 负责
- 中断后的业务通知目前尚未大量展开，后续可按“直接回调模块”或“任务通知/事件标志”两种模式扩展

### 8.2 FreeRTOS 诊断钩子

当前已在 `bsp/core/freertos_hooks.c` 中实现：

- `vAssertCalled()`
- `vApplicationStackOverflowHook()`
- `vApplicationMallocFailedHook()`

诊断策略固定为：

- 记录 panic 原因、文件、行号、任务名
- 关闭中断
- 陷入死循环，便于调试器定位

## 9. CMake 与构建链路现状

当前 `project/CMakeLists.txt` 已完成以下能力：

- 递归收集 `app/`、`module/`、`bsp/` 源码
- 编译 `Libraries/hal` 中的 HAL 源码
- 编译 FreeRTOS 内核所需最小文件集
- 编译系统文件与启动文件
- 使用 `project/STM32G431RBTX_FLASH.ld` 链接
- 构建后输出 `elf / hex / bin / map`

当前构建链路已经验证通过，FreeRTOS 相关符号已能正常链接进最终产物。

## 10. 当前限制与下一步建议

当前工程虽然已完成 RTOS 底座接入，但仍属于“平台优先”的首版：

- `module_motor_process()` 尚未接入任务调度
- `bsp/driver/` 下大多数驱动仍是空实现
- `protocol`、`fault` 等业务模块尚未建立
- 当前还没有引入任务通知、队列、信号量等实际业务同步机制

建议的后续演进顺序如下：

1. 在 `bsp/driver/` 中补齐 PWM、ADC、TIM、UART 等实际接口
2. 将 `motor` 任务接入真实的控制数据流
3. 按业务需要新增 `protocol` / `fault` 等模块
4. 根据实时性需求决定控制路径是继续用周期任务，还是切到“中断 + 通知”的混合模式

---

## 11. 默认假设

- 当前系统继续使用 `HSI 16 MHz`，不启用 `PLL`
- HAL 与 FreeRTOS 共用 `SysTick`
- `Libraries/` 保持第三方代码定位，不放用户业务代码
- 目标项目仍然是面向 BLDC 场景继续扩展的 STM32G431 工程
